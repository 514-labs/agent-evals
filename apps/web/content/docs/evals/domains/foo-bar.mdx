The Foo Bar domain provides dummy data with no production semantics. Use it to validate eval infrastructure, test assertion frameworks, and prototype new scenarios before committing to a real domain.

## Core Data

- Synthetic dimension tables with deterministic row counts.
- Generated event streams with configurable cardinality.
- Placeholder metric columns (integers, floats, timestamps).
- Simple foreign-key relationships for join validation.

## Characteristic Challenges

- Data carries no business meaning -- assertions must rely on structural checks rather than domain logic.
- Useful for verifying pipeline plumbing, schema propagation, and idempotency without domain expertise.
- Seed data should be small enough to run in seconds but large enough to exercise partitioning and deduplication paths.

## Evaluation Focus

- Confirm that the full create-build-run-score loop works end to end.
- Test gate assertion discovery, core assertion wiring, and JSON output formatting.
- Validate harness install scripts and service health checks in isolation.
- Prototype new competency or feature assertions before porting to a real domain.

## Typical Outputs

- Row-count and checksum assertions over synthetic tables.
- Schema-shape validation (column names, types, nullability).
- Pipeline runtime and idempotency checks with trivial data volumes.
- Scaffold-quality scoring output used to verify leaderboard ingestion.
