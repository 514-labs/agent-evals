---
title: "Realtime Feeds"
description: "Evaluate streaming data delivery for correctness, resilience, and latency commitments."
---

Realtime feeds evaluate whether agents can build streaming data products that remain correct under delay, disorder, and failure.

## Feature Intent

Consumers should receive low-latency updates with clear guarantees on correctness and timeliness.

## Capability Boundaries

Qualifies:

- Handles event time vs processing time intentionally
- Defines late-data and out-of-order policies
- Provides explicit freshness and latency objectives
- Supports recovery from failures without silent data corruption

Does not qualify:

- Stream demos with no durability or replay behavior
- Latency claims without observable SLI/SLO evidence
- Pipelines that duplicate, drop, or reorder records without policy

## Competency Dimensions

- Event-time correctness and ordering guarantees
- Latency/freshness objectives and observability
- Fault tolerance, replay, and recovery behavior
- Backpressure and throughput scalability
- Operability and incident response readiness

## Level Rubric (L1-L5)

| Level | Expected Capability |
| --- | --- |
| L1 | Streams data with best-effort behavior and weak guarantees under failure. |
| L2 | Provides basic continuous updates but limited control over disorder and replay. |
| L3 | Implements reliable stream semantics with documented lateness policy and measurable latency targets. |
| L4 | Handles high load and partial failures with robust recovery, observability, and consistent outputs. |
| L5 | Demonstrates mature stream operations with policy-driven correctness guarantees and predictable performance at scale. |

## Required Evidence

- Stream contract describing ordering, watermarking, and lateness handling
- Latency and freshness dashboards with percentile distributions
- Failure/recovery test results (node loss, broker restart, replay)
- Throughput test evidence under peak and burst load
- On-call runbooks for degraded stream conditions

## Failure Modes and Anti-Patterns

- Uses processing time where event time is required for correctness
- No defined behavior for late-arriving events
- Backpressure cascades causing unstable latency and dropped records
- Recovery process replays inconsistently and produces duplicate outcomes

## Suggested Test Scenarios

- Inject delayed and out-of-order events and validate aggregation correctness
- Simulate consumer or processor crash and verify recovery guarantees
- Stress burst ingestion and monitor backpressure propagation
- Compare replayed results with original run for deterministic consistency
