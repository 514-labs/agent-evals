---
title: "Metrics Layers"
description: "Evaluate how agents define, govern, and serve consistent business metrics."
---

Reporting metrics layers evaluate whether agents can create reusable metric definitions that stay consistent across BI tools, exports, APIs, and applications.

## Feature Intent

Users should get the same answer for the same business question, independent of where the metric is consumed.

## Capability Boundaries

Qualifies:

- Centralizes metric logic with clear measures, dimensions, and entities
- Exposes governed definitions to downstream consumers
- Tracks lineage and change impact before release

Does not qualify:

- KPI logic duplicated independently in every dashboard
- Metrics with no owner, no contract, or no test coverage
- Ad hoc SQL outputs treated as canonical reporting sources

## Competency Dimensions

- Semantic consistency across all consumers
- Governance and ownership of metric contracts
- Lineage visibility and impact analysis
- Change safety (tests, CI checks, and version discipline)
- Query usability for both technical and non-technical consumers

## Level Rubric (L1-L5)

| Level | Expected Capability |
| --- | --- |
| L1 | Defines isolated metrics in individual reports with little reuse and no central governance. |
| L2 | Introduces shared metric logic for a subset of reports but inconsistencies remain across tools. |
| L3 | Establishes a governed metrics layer with reusable dimensions and basic lineage/testing. |
| L4 | Operates robust semantic contracts with CI validation, impact previews, and controlled rollout. |
| L5 | Maintains enterprise-grade semantic governance with strong discoverability, versioning, and high consumer trust. |

## Required Evidence

- Canonical metric specs (definition, grain, and owners)
- Compatibility matrix showing where each metric is consumed
- Lineage and dependency graph for key measures and dimensions
- Test suite covering metric correctness and dimensional behavior
- Change management process (review, rollout, and rollback)

## Failure Modes and Anti-Patterns

- Similar metric names with different business semantics
- Hidden joins or filters that alter metric meaning by tool
- Unversioned metric changes that silently break downstream reports
- Semantic layer performance issues from poorly modeled entities

## Suggested Test Scenarios

- Introduce a metric definition change and validate impact reports before merge
- Query one metric from multiple interfaces and compare outputs
- Add a new dimension and verify no regression in existing consumers
- Remove a deprecated metric and confirm safe fallback behavior
