# Performance Dashboards

Performance dashboards measure whether an agent can build and maintain dashboards that are trusted for decision-making, not just visually complete.

## Feature Intent

Users should be able to monitor business and system performance with:

- Accurate metric values
- Clear dimensional drill paths
- Predictable freshness and latency
- Stable behavior as data volume and query load grow

## Capability Boundaries

Qualifies:

- Builds dashboards from governed metric definitions
- Declares freshness and latency expectations
- Supports useful slice-and-dice paths by business dimensions
- Handles failures and stale states explicitly

Does not qualify:

- Dashboard mockups without production data logic
- Hardcoded numbers without traceable metric definitions
- Dashboards with no freshness or data quality signaling

## Competency Dimensions

- Metric correctness and semantic consistency
- Data freshness, latency, and staleness handling
- Dimensional modeling and drill-path coherence
- Performance and scalability under realistic workload
- Operability (alerts, ownership, and runbook clarity)

## Level Rubric (L1-L5)

| Level | Expected Capability |
| --- | --- |
| L1 | Produces static or minimally dynamic dashboard views with weak metric traceability. |
| L2 | Connects live data and basic filters but has limited freshness guarantees and weak failure handling. |
| L3 | Delivers governed KPIs, clear dimensions, and explicit freshness indicators with acceptable performance. |
| L4 | Supports high-confidence decisions with resilient refresh logic, scalable query patterns, and robust alerting. |
| L5 | Demonstrates mature data-product behavior: proactive anomaly signaling, clear ownership, and reliable multi-team usage at scale. |

## Required Evidence

- Metric definitions and lineage to source models
- Refresh schedules, SLAs/SLOs, and staleness behavior
- Query performance evidence (load tests or observed latency percentiles)
- Data quality checks tied to dashboard critical metrics
- Runbook and ownership metadata for incidents

## Failure Modes and Anti-Patterns

- Conflicting KPI definitions across views
- Hidden aggregation changes that alter trends silently
- Slow dashboards caused by unbounded queries or poor partition pruning
- Missing stale-data indicators that create false confidence

## Suggested Test Scenarios

- Add a new dimension and verify historical metric stability
- Introduce delayed upstream data and verify stale-state handling
- Stress test concurrent filters and drilldowns for latency regression
- Simulate a failed refresh and verify alerting and operator guidance
